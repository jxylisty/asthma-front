
核心功能模块详细设计

1. 门户与智能检索系统（首页）

1.1 动态三维视觉背景与全局状态监控条

●交互与视觉（UI/UX）：

○页面采用非静态背景，通过 particles.js 渲染出具备伪随机连线效果的 3D 拓扑粒子网。当鼠标在屏幕上移动时，粒子网会产生微弱的引力或斥力扰动，隐喻“化合物与生物靶点的动态网络绑定”。

○搜索框下方并排设计 3 个极简科技感卡片，数字部分使用 countUp.js 实现页面加载时的从 0 到真实数据的滚动动画。

●业务逻辑（Business Logic）：

○前端挂载（onMounted）时，向后端发送异步请求 GET /api/system/metrics。

○后端返回字段：

■total_prescriptions：系统目前收录的经典方剂总数（如 128 首）。

■total_compounds：底层多模态特征库中化合物的总数量（如 24,531 种）。

■total_targets：系统关联的哮喘/呼吸系统靶点基因总数（如 1,422 个）。

1.2 智能检索中枢 (Smart Search Console)

●交互与视觉（UI/UX）：

○位于页面正中央。输入框内置“语音输入”图标（扩展内容）。

○支持输入框激活时，下方自动弹窗展示“热门经典方剂推荐”标签（如：小青龙汤、定喘汤、射干麻黄汤、麻杏石甘汤）。

●业务逻辑（Business Logic）：

○多维模糊匹配：用户输入时，前端触发防抖机制（Debounce, 延时 300ms 发送请求），调用后端 GET /api/search/suggest?keyword=xxx。

○后端匹配算法：后端接收到 keyword 后，需进行多重条件检索：

1.匹配方剂的标准名称（如“小青龙汤”）。

2.匹配方剂的拼音首字母（如输入“xql” -> 匹配 xiaoqinglongtang）。

3.匹配方剂包含的单味药材（如输入“麻黄” -> 匹配所有含麻黄的方剂）。

4.匹配方剂对应的中医证型（如输入“外寒” -> 匹配“外寒内饮证”的方剂）。

○实体对齐与纠错：若用户输入存在轻微错别字（如“小青良汤”），后端引入编辑距离算法（Levenshtein Distance），在下拉联想框中纠正并提示“您是不是要找：小青龙汤”。

●状态模拟设计（UX剧场）：

○当用户点击搜索或回车时，页面不直接跳转，而是弹出一个高科技感的“数据分析中”全屏遮罩层。

○遮罩层内渲染一个类似细胞膜吸收分子的 3D 循环动画，并伴随真实时间步长（共持续 1.5 秒）动态滚动显示计算日志：

■[0.0s - 0.4s]：“正在加载多模态分子特征向量...”

■[0.4s - 1.0s]：“正在启动 PU 学习算法预测入血概率...”

■[1.0s - 1.5s]：“正在完成靶点通路映射，生成拓扑图谱...”

○日志结束后，丝滑路由跳转至详情页。（注：此设计能极大增强评委对“后台有硬核算法在实时运行”的直观感知）

2. 多维药理可视化看板（大众科普态）

2.1 方剂全景信息卡 (Prescription Overview Card)

●交互与视觉（UI/UX）：

○位于详情页左上角，采用国风与科技感结合的卡片设计（如淡木纹背景配科技蓝边框）。

●业务逻辑（Business Logic）：

○展示该方剂的基本档案：方剂名、出处（如《伤寒论》）、标准药材组成。

○创新点：针对每味药材，鼠标悬停时弹窗显示该药材在系统里的理化画像（如麻黄：性温、味辛、微苦，归肺、膀胱经；主要功效为发汗散寒、宣肺平喘）。

2.2 吸收效能分析组件 (Absorption Prediction Component)

●交互与视觉（UI/UX）：

○位于详情页右上角。不要枯燥的表格，采用类似游戏“角色属性面板”的设计。

○只筛选出该方剂中入血预测概率前 5 名的核心化合物，每种化合物占用一行。

○每一行左边是化合物名称，中间是醒目的渐变色能量条（根据概率值，90%以上为红色，80%~90%为橙色），右边是数字（如 98.2%）。

●业务逻辑（Business Logic）：

○数据来源：调用后端 GET /api/prescriptions/{name}/components 接口。

○这些数据是你们的 PU 学习模型（多模态特征输入） 跑出来的真实概率值。

○大白话翻译机制：每一行下方用小字附带一句徐晨洋（数据）整理的科普说明。例如：

■化合物：麻黄碱 (Ephedrine) -> “【预测入血概率 98.2% (极高)】：本方核心活性成分，能直接作用于支气管平滑肌，具有强效的舒张气道、缓解哮喘急性发作的作用。”

2.3 多维干预雷达图 (Efficacy Radar Chart)

●交互与视觉（UI/UX）：

○位于详情页左下方，利用 ECharts 渲染一个多维雷达图。

○定死 4 个核心疗效维度：抗炎指数、平喘（支气管舒张）指数、免疫调节指数、气道修复指数。

●业务逻辑（Business Logic）：

○指数怎么来的？（核心逻辑，必须跟组员对齐）：

1.后端数据库存储了每个化合物对应的“基因靶点（Targets）”（通过网络药理学预测或数据库检索得出）。

2.这些靶点会富集到不同的生物学通路（KEGG Pathway）上。

3.系统建立映射字典：

■若靶点富集在 TNF signaling pathway、IL-17 signaling pathway 等炎症通路上，则归为 抗炎类靶点。

■若靶点富集在 Calcium signaling pathway（调节平滑肌舒张）通路上，则归为 平喘类靶点。

■若靶点富集在 T cell receptor signaling pathway 等通路上，则归为 免疫调节类靶点。

4.计算公式：$\text{某项指数} = (\text{该方剂入血成分中所命中该通路的有效靶点数} / \text{该通路哮喘核心已知总靶点数}) \times 100\%$。

○用户解释：雷达图右上角必须保留一个 “❓” 悬浮图标，鼠标移上去显示这套计算公式的严谨学术解释，让评委挑不出毛病。  这个部分不确定 根据实际数据待定

3. 多模态预测与网络药理学工作台（专家科研态）

3.1 核心算法性能面板 (Model Metrics)

●交互与视觉（UI/UX）：

○点击“进入专家模式”后切换至该面板。界面转为深色系（Dark Mode），营造实验室终端的黑客既视感。

●业务逻辑（Business Logic）：

○图表 A：ROC / PR 曲线对比图：ECharts 折线图。展示你们的 PU 学习算法 在标准中药数据集上的表现，并绘制出传统监督学习（如纯 SVM、随机森林）在缺乏负样本时的传统表现曲线。两条线一对比，直观展示你们算法在“解决中药无负样本标注痛点”上的巨大优势。

○图表 B：多模态特征权重热力图 (Feature Weight Heatmap)：展示一幅热力矩阵。横轴是化合物的多模态特征（如：分子量 MW、脂水分配系数 LogP、可旋转键数量、氢键供体数、拓扑极性表面积 TPSA 等），颜色深浅代表该特征对 PU 学习模型判断“该成分能否入血”的贡献度。证明你们做了扎实的数据工程。



3.2 高维靶点拓扑网络 (Topological Network Viewer)



由“首页搜索”决定（查特定方剂）

这是最常规的用户路径。



用户动作：用户在首页搜索了“小青龙汤”，然后一路点进了“专家模式页”。

图表内容：这时候，画板上只画“小青龙汤”这一首方剂的网络图。

数据流转（给后端李锦程的逻辑）：

前端知道当前页面属于“小青龙汤”，于是向后端发送请求：GET /api/network?prescription=小青龙汤

后端去数据库里查：小青龙汤包含哪几味药？这几味药通过 PU 学习筛选出了哪几个高概率入血成分？这些成分又对应了哪些哮喘靶点？

后端把这批专属数据打包成 JSON 发给前端，前端 Cytoscape 渲染出来。

展示效果：最顶层是1个方剂节点（小青龙汤） -> 往下连着 8个药材节点 -> 再往下连着 30个成分节点 -> 最底层连着 50个靶点节点。

●交互与视觉（UI/UX）：

○占据专家页的核心正中央区域，使用 Cytoscape.js 渲染。

●业务逻辑（Business Logic）：

○图元视觉编码：

■节点（Nodes）：方剂（金黄色五角星）、单味药材（绿色方块）、化合物成分（蓝色圆形，大小/透明度由 PU 学习入血概率决定）、哮喘靶点基因（红色菱形）。

■连线（Edges）：药材-成分（灰色实线）、成分-靶点（半透明科技蓝发光线，粗细代表结合能/活性值）。

○极客级交互：

1.一阶邻居高亮：鼠标点击任意一个“靶点基因（如 IL-4）”，网络中其他不相关的几百个节点瞬间变透明，只有跟 IL-4 连结的入血化合物和药材保持高亮，清晰展示药理路径。

2.右键菜单：右键点击化合物节点，可弹出小窗查看该化合物的标准 SMILES 2D 结构式图像（由后端通过 rdkit 库动态生成）。

3.网络导出：右上角提供“导出为 SVG/PNG”按钮，方便科研人员直接把图下载下来贴进论文里。

3.3 动态组方预测沙盒 (Dynamic Prescription Sandbox) —— 

●交互与视觉（UI/UX）：

○专家页左侧的工作台面板，包含一个“自定义配方操纵杆”。

●业务逻辑（Business Logic）：

○药材选择器（防错实体对齐）：

■这里是一个 el-select 远程搜索下拉框。绝不允许用户自由手打输入文本，防止出现“双花”、“金银花”、“忍冬藤”等由于中药俗名、别名、炮制方法不同导致后台数据库无法识别的灾难。

■后端维护一张 alias_mapping 字典表。无论用户搜“双花”还是“金银花”，下拉列表均规范化显示并输出标准 ID：[ID: 10023] 金银花 (Flos Lonicerae)。

○动态计算工作流（Real-time Inference Pipeline）：

1.用户在沙盒里选了三味药：麻黄 + 杏仁 + 石膏，并点击“运行 AI 实时推理”按钮。

2.前端向后端发起 POST /api/predict/sandbox，携带参数 {"herb_ids": [1001, 1005, 1012]}。

3.后端算法实时响应：

■Step 1：去底层化合物特征库中，把这三味药包含的所有化合物（假设共 300 个）的理化特征向量调出来。

■Step 2：将这 300 个特征矩阵直接喂给已经训练好的 PU 学习模型（FastAPI 异步调用 Python 模型推理）。

■Step 3：模型在 200ms 内给出这 300 个化合物各自的入血概率，过滤掉概率低于 70% 的，留下 15 个核心入血成分。

■Step 4：将这 15 个成分去比对哮喘靶点库，计算出新的拓扑连线数据和 KEGG 雷达图指数。

4.前端接收到全新的 JSON 数据，动态重绘 Cytoscape.js 拓扑网络图，并动态刷新疗效雷达图。
